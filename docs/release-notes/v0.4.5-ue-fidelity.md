# v0.4.5 Release Notes - UE Fidelity Improvements

**Release Date:** 2026-06-09  
**Version:** 0.4.5  
**Branch:** 0.4.5-dev

## Overview

v0.4.5 实现了 6 项关键的 UE 保真度改进，使 uasset_read 的解析行为更接近 Unreal Engine 的 FLinkerLoad 生命周期。这些改进解决了之前审计中发现的核心问题，将解析器从"容错提取器"提升为"UE 等价加载器"。

## Breaking Changes

### 1. 状态模型统一 (Status Model)

**之前:** `success | fail | error`  
**之后:** `success | partial | failed`

- `success`: 无错误，所有 export 解析成功
- `partial`: 部分错误或某些 export 为 opaque/skipped，但核心数据可用
- `failed`: 严重错误，无可用数据

**影响:**
- `ParseResult.status` 属性现在返回新的三态值
- `build_status_info()` 函数适配新模型
- 所有使用 `result.status` 的代码需要更新检查逻辑

**迁移示例:**
```python
# 旧代码
if result.status == 'fail':
    handle_error()

# 新代码
if result.status in ('partial', 'failed'):
    handle_error()
```

### 2. Payload 偏移策略 (Payload Offset Strategy)

**之前:** UE5.10+ 默认使用 `ScriptSerializationStartOffset/EndOffset`  
**之后:** 默认使用 `SerialOffset/SerialSize`（与 UE LinkerLoad.cpp:4793 对齐）

**影响:**
- 属性解析现在从 `SerialOffset` 开始，而不是 `ScriptSerializationStartOffset`
- ScriptSerialization 偏移保留为诊断字段（`_script_serialization_start_absolute`）
- 对于包含 class-specific payload 的资产（如 StaticMesh、Texture2D），现在会解析更多数据

**迁移:** 无需代码修改，但解析结果可能包含更多属性。

## New Features

### 1. UE 风格加载生命周期 (UE-Style Loading Lifecycle)

**问题:** 之前 `post_load()` 在 export 属性解析之前调用，导致 `_resolve_property_references()` 无法看到已预加载的对象。

**解决方案:**
- 将 `linker.post_load()` 移到所有 export 预加载之后
- 使用 `linker.preload(idx)` 替代直接调用 `parse_properties_from_export`
- 确保 archive 在 preload 期间保持打开

**执行顺序:**
```
link() → preload(idx) × N → post_load()
```

**影响:** ObjectProperty 引用现在能正确解析为 UObjectInstance 对象。

### 2. 类序列化策略表 (Class Serialization Strategy Table)

**问题:** StaticMesh、Texture2D 等类有专用的 `Serialize()` 方法，但我们没有实现，导致它们被错误地标记为"success"。

**解决方案:**
- 新建 `class_serialization_strategy.py` 模块
- 定义 4 种策略：`FULL_SERIALIZER`、`TAGGED_PROPERTIES_ONLY`、`OPAQUE_CLASS_PAYLOAD`、`SKIP_UNSUPPORTED`
- 在 `linker.preload()` 中检查策略，提前拦截不支持的类

**策略映射示例:**
- `BlueprintGeneratedClass` → `TAGGED_PROPERTIES_ONLY`（可解析）
- `StaticMesh` → `OPAQUE_CLASS_PAYLOAD`（标记为 opaque）
- `NiagaraSystem` → `SKIP_UNSUPPORTED`（跳过）

**影响:** 不支持的类现在正确标记为 `opaque` 或 `skipped`，而不是 `success`。

### 3. SoftObjectPath 索引化解析 (SoftObjectPath Index-Based Resolution)

**问题:** UE5.7+ 中，当存在 `SoftObjectPathList` 时，`SoftObjectProperty` 应该读取 int32 索引，而不是 FString 对。

**解决方案:**
- 在 `ParseResult` 中存储 `soft_object_path_list`
- 修改 `parse_soft_object_property()` 支持双模式：索引/字符串
- 越界索引产生诊断信息，不崩溃

**影响:** UE5.7+ 资产的软引用现在能正确解析。

### 4. DependsMap FPackageIndex 语义 (DependsMap FPackageIndex Semantics)

**问题:** DependsMap 值被错误地当作 export index 处理，而不是 FPackageIndex。

**解决方案:**
- 修改 `linker._build_dependency_graph()` 使用 FPackageIndex 语义
- 正值 = export (1-based)
- 负值 = import (-1-based)
- 零 = null
- 使用 `resolve_package_index()` 正确解析导入/导出依赖

**影响:** 依赖图现在能正确引用 import 对象。

### 5. 统一状态模型 (Unified Status Model)

**问题:** 不同输出入口的状态不一致（JSON 用 `success/fail/error`，IR 用 `success/partial/failed`）。

**解决方案:**
- 统一使用 `success | partial | failed`
- `ParseResult.status` 属性动态计算
- `LinkerParseResult` 也实现相同的 status 属性

**影响:** 所有输出格式（JSON、文本、IR）的状态一致。

## Bug Fixes

### 1. Archive 生命周期修复

**问题:** `parse_uasset_with_linker(preload_all=True)` 失败，因为 archive 在 `_parse_package_core` 的 finally 块中关闭，但 `preload_all` 在之后执行。

**解决方案:** 重构生命周期，确保所有 preload 操作在 archive 关闭前完成。

### 2. StaticMesh Opaque 标记修复

**问题:** `linker.preload()` 将 opaque 状态设置到 `UObjectInstance`，但未同步到 `export` 对象。

**解决方案:** 在 `linker.preload()` 中同时设置 `instance` 和 `export` 的 `parse_status` 和 `fallback_reason`。

### 3. LinkerParseResult Status 属性缺失

**问题:** `LinkerParseResult` 没有 `status` 属性，导致集成测试失败。

**解决方案:** 为 `LinkerParseResult` 添加与 `ParseResult` 相同的 `status` 属性实现。

## Known Limitations

1. **Opaque 类仍然无法完全解析:** StaticMesh、Texture2D 等类标记为 `opaque`，表示我们有它们的元数据，但没有完整的 `Serialize()` 实现。

2. **SoftObjectPath 索引越界:** 当索引超出 `soft_object_path_list` 范围时，返回空值并记录诊断，不抛出异常。

3. **DependsMap 循环依赖:** 当前实现不检测循环依赖，但这不影响解析正确性。

## Future Work

1. **实现更多类的 Serialize():** 为 StaticMesh、Texture2D 等高频资产类型实现完整的序列化逻辑。

2. **循环依赖检测:** 在依赖图构建后添加循环检测。

3. **性能优化:** 对于大型资产（>10000 个 export），考虑延迟创建 UObjectInstance。
