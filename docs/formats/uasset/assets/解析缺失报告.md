# UE 蓝图解析缺失报告

## 概览

本报告记录 BP_FirstPersonCharacter.uasset (UE5.17) 解析过程中发现的格式细节缺失和实现缺口。

**文件信息：**
- 资产路径：`E:\Develop\lib\UnrealEngine\Samples\FirstPerson\Content\FirstPerson\Blueprints\BP_FirstPersonCharacter.uasset`
- UE 版本：UE5 = 1017 (OS_SUB_OBJECT_SHADOW_SERIALIZATION)
- 交易版本： cooked

---

## 1. Pin 连接序列化格式缺失

### 1.1 问题描述

UE5.7 的 `UEdGraphPin` 序列化格式在官方文档中缺乏详细说明，特别是 **PinReference Header** 格式。

**当前实现：**
```
// D-12: UE5 Pin array uses PinReference format with external header
Header: b_null_ptr (4B) + owning_node (4B) + pin_guid (16B) = 24B
Body: Complete UEdGraphPin (duplicates owning_node + pin_guid)
```

**问题字段：**

| 字段 | 类型 | 位置 | 状态 | 说明 |
|------|------|------|------|------|
| `SourceIndex` | int32 | body | 部分实现 | UE5 新增，文档未明确 |
| `PersistentGuid` | FGuid | body | 实现 | 编辑器专用，文档缺少版本判断 |
| `BitField` | uint32 | body | 实现 | 位域定义缺少字段名映射 |

### 1.2 引用缺失

- `FBlueprintEditorUtils::FPinReference` 序列化运算符缺乏源码路径引用
- `UEdGraphPin::Serialize()` 的完整序列化顺序缺少版本兼容性说明

### 1.3 解析问题

```
LinkedTo read failed at pos 115013: Pin array count 779314540 exceeds MAX_LINKEDTO_PER_PIN 100
LinkedTo read failed at pos 119045: Pin array count 1886220099 exceeds MAX_LINKEDTO_PER_PIN 100
LinkedTo read failed at pos 128834: Pin array count -1862270976 exceeds MAX_LINKEDTO_PER_PIN 100
```

**现有恢复机制：**
- Phase 73: 滑动窗口恢复 (scan_window=8)
- Phase 73: SubPins 重同步尝试

**缺失内容：**
- PinReference header 的 null 语义没有明确版本定义
- count 异常时的回退策略缺少源码依据

## 2. FText/FSstring 序列化格式缺失

### 2.1 FText 历史类型

当前实现有 `read_ftext_with_history()` 函数，但文档缺失：

| history_type | 类型 | 文档状态 |
|--------------|------|----------|
| -1 (0xFF) | None | 部分实现 |
| 0 | Base | 部分实现 |
| 1 | NamedFormat | 部分实现 |
| 2-10 | OrderedFormat/ArgumentFormat/AsNumber/... | 缺失 |

**问题：**
- 大多数 history_type 仅"untolerant mode"下抛出异常
- 缺少 C++ 源码路径引用（Text.cpp L850-1044）

### 2.2 FString 和 FText 的错误使用

```
PinToolTip — FString (NOT FText!)
DefaultTextValue — FText (NOT FString!)
```

**当前实现：** 已区分，但文档中缺少对比说明。

## 4. PropertyTag 扩展格式

### 4.1 UE5 >= 1011 属性标签扩展

**当前实现：**
```python
# L1515-1519: Script Serialization Control
if summary.file_version_ue5 >= 1011:
    ctrl = archive.read_u8()
    if ctrl & 0x02:
        archive.read_u8()
```

**缺失内容：**
- `ctrl & 0x01` 的含义（可覆盖序列化标志）
- `ctrl & 0x02` 的具体作用（序列化控制扩展）
- 完整的 FPropertyTag 扩展字段表

### 4.2 PropertyTag 完整格式

当前仅有精简定义，缺少：

| 字段 | UE4 | UE5 | 说明 |
|------|-----|-----|------|
| `Size` | int32 | - | UE5 使用 ValueEndOffset |
| `ValueEndOffset` | - | int32 | UE5 新增，替代 Size |
| `Type` | int32 | int32 | 属性类型枚举 |
| `SerialType` | - | int32 | UE5.11 新增 |
| `HasUnwantedProp` | - | int8 | UE5.12 新增 |

## 5. EdGraph 节点数组格式

### 5.1 脚本序列化边界

**当前实现：**
```python
# L1659: End marker + pins_count
pins_offset = script_serial_offset + script_serial_size + 4  # Skip end marker
archive.seek(serial_offset + pins_offset)
pins_count = archive.read_i32()
```

**问题：**
- `script_serial_size` 的来源不明确（PropertyTag 解析得出还是 Export 表字段）
- 结束标记 (end marker) 的值没有验证

### 5.2 nodes_count 不完整问题

**代码：**
```python
# L1778-1812: UE 5.x fallback
# Main path nodes_count can be incomplete due to UE5 serialization differences
```

**缺失：**
- 何时触发 fallback 的明确条件
- 主路径与 fallback 的解析差异具体原因


```
16. ref_pass_through (PinReference)
17. persistent_guid (FGuid)  -- EditorOnly
18. bitfield (uint32)        -- EditorOnly
```

**问题：**
- 部分字段顺序 undocumented
- EditorOnly 字段的版本判断缺失

## 9. 解析恢复机制不完整

### 9.1 LinkedTo 恢复

**现有策略：**
```
1. 滑动窗口扫描 (±8 bytes)
2. 候选 count 验证
3. PinReference 结构校验
4. 低置信度过滤
```

**缺失：**
- 各种异常模式的处理优先级
- 恢复失败后的备用策略
- 多次恢复尝试的回退逻辑

### 9.2 SubPins 重同步

**问题：**
- SubPins 与 LinkedTo 的分界点不明确
- 恢复位置验证不足
- 与主解析路径的融合策略
